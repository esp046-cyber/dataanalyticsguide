üîç Problem 1: Missing Data (NULL Values)

Types of Missing Data

MCAR (Missing Completely At Random):
  • Example: Survey respondent accidentally skipped question
  • Safe to delete or impute without bias
MAR (Missing At Random):
  • Example: Younger customers more likely to skip income question
  • Pattern exists but can be accounted for in analysis
MNAR (Missing Not At Random):
  • Example: High earners deliberately skip income question
  • Most dangerous - creates bias if not handled properly

Detection Methods

Python: df.isnull().sum() SQL: WHERE column IS NULL Excel: =COUNTBLANK(range)
Python Example:
# Check missing percentage
missing = df.isnull().sum() / len(df) * 100
print(missing[missing > 0].sort_values(ascending=False))

Handling Strategies

DELETE
Remove rows: Use when < 5% data missing and MCAR. Risk: lose valuable data, reduce sample size.
MEAN/MEDIAN
Impute center: Use median for skewed data, mean for normal. Risk: reduces variance, masks patterns.
MODE/FORWARD FILL
Categorical/Time series: Most frequent value or carry forward last known. Risk: creates artificial patterns.
PREDICTIVE
ML imputation: Use KNN, MICE, or regression to predict missing values. Risk: complex, can overfit.
FLAG & ANALYZE
Create indicator: Keep nulls, add "is_missing" column, analyze as separate segment. Best practice for MNAR.
Decision Tree:
‚Ä¢ Missing > 40%? ‚Üí Drop column or investigate data source
‚Ä¢ Missing 5-40%? ‚Üí Impute or flag
‚Ä¢ Missing < 5% and MCAR? ‚Üí Safe to delete rows
‚Ä¢ Time series? ‚Üí Forward/backward fill
‚Ä¢ Categorical? ‚Üí Create "Unknown" category

üîÑ Problem 2: Duplicate Records

Types of Duplicates

Exact Duplicates:
  • Every column identical - often from data extraction errors or double-clicking "submit"
  • Solution: Keep first occurrence, drop rest
  • Python: df.drop_duplicates()
Partial Duplicates:
  • Same key fields (customer_id, order_id) but different timestamps or details
  • Example: Customer record updated multiple times in database
  • Solution: Keep most recent or aggregate
  • Python: df.sort_values('timestamp').drop_duplicates(subset=['customer_id'], keep='last')
Fuzzy Duplicates:
  • Similar but not exact: "John Smith" vs "J. Smith", "Apple Inc." vs "Apple Inc"
  • Solution: Standardize first, then deduplicate
  • Python: df['name'] = df['name'].str.lower().str.strip()

Detection Techniques

SQL Detection:
-- Find duplicate order_ids
SELECT order_id, COUNT(*) as count
FROM orders
GROUP BY order_id
HAVING COUNT(*) > 1
ORDER BY count DESC;
Python Advanced:
# Find duplicates across multiple columns
dupes = df[df.duplicated(subset=['email', 'phone'], keep=False)]
print(f"Found {len(dupes)} duplicate records")

Business Rules for Handling

TRANSACTION DATA
Keep all with same ID: Multiple orders from same customer are valid, not duplicates.
MASTER DATA
Keep most recent: Customer/product tables should have one record per ID (latest version).
AGGREGATE
Merge intelligently: Sum transactions, keep max timestamp, concatenate notes with separator.

üìä Problem 3: Data Type Issues

Common Mismatches

Dates as Strings:
  • Problem: "2024-01-15" stored as text, can't do date math or filtering
  • Python: df['date'] = pd.to_datetime(df['date'])
  • SQL: CAST(date_column AS DATE)
  • Watch for: Mixed formats ("01/15/2024" vs "2024-01-15")
Numbers as Strings:
  • Problem: "1,234.56" with commas, currency symbols "$100", or leading zeros "00123"
  • Python: df['price'] = df['price'].str.replace(',','').str.replace('$','').astype(float)
  • Excel: Use SUBSTITUTE() and VALUE() functions
Mixed Types in Column:
  • Problem: Column has "N/A", "None", 0, and actual numbers
  • Solution: Replace text nulls first, then convert: df['col'].replace(['N/A','None'], np.nan).astype(float)
Boolean Confusion:
  • Problem: TRUE/True/1/Y/yes all mean the same thing but stored differently
  • Solution: Standardize: df['flag'] = df['flag'].map({'yes':1, 'Y':1, 'TRUE':1, 'no':0, 'N':0, 'FALSE':0})

üéØ Problem 4: Outliers & Anomalies

Detection Methods

VISUAL
Box plots, histograms: Spot values far from distribution. Quick but subjective.
IQR METHOD
Interquartile Range: Outlier if < Q1-1.5√óIQR or > Q3+1.5√óIQR. Standard statistical approach.
Z-SCORE
Standard deviations: Outlier if |z-score| > 3. Assumes normal distribution.
ISOLATION FOREST
ML-based: Detect anomalies in multi-dimensional data. Best for complex patterns.
Python IQR Example:
Q1 = df['price'].quantile(0.25)
Q3 = df['price'].quantile(0.75)
IQR = Q3 - Q1
outliers = df[(df['price'] < Q1-1.5*IQR) | (df['price'] > Q3+1.5*IQR)]
print(f"Found {len(outliers)} outliers")

Handling Strategies

Investigate First:
  • Is it real? Example: $10M sale might be legitimate enterprise deal
  • Is it data error? Example: Age = 150 years, negative revenue
  • Is it measurement error? Example: Weight in lbs recorded as kg
Treatment Options:
  • Keep: If legitimate and analysis-relevant (e.g., studying high-value customers)
  • Delete: If clear error and < 1% of data
  • Cap: Winsorize at 95th/99th percentile to reduce impact
  • Transform: Log transform to reduce skew from outliers
  • Segment: Analyze outliers separately (e.g., VIP customer segment)

üìù Problem 5: Inconsistent Formatting

Text Data Issues

Case Sensitivity:
  • Problem: "apple", "Apple", "APPLE" treated as different categories
  • Solution: df['product'] = df['product'].str.lower()
Whitespace:
  • Problem: Leading/trailing spaces, multiple spaces between words
  • Solution: df['name'] = df['name'].str.strip().str.replace('\s+', ' ', regex=True)
Special Characters:
  • Problem: "Caf√©" vs "Cafe", "O'Brien" vs "O Brien"
  • Solution: Normalize unicode or remove: import unicodedata; unicodedata.normalize('NFKD', text)
Abbreviations & Variations:
  • Problem: "California" vs "CA" vs "Calif.", "Street" vs "St." vs "St"
  • Solution: Create mapping dictionary or use fuzzy matching libraries (fuzzywuzzy)

Date Format Nightmares

Common Chaos:
‚Ä¢ MM/DD/YYYY vs DD/MM/YYYY (is 01/02/2024 Jan 2 or Feb 1?)
‚Ä¢ Excel date serial numbers (44927 = Dec 31, 2022)
‚Ä¢ Text dates: "January 15, 2024" vs "15-Jan-24"
‚Ä¢ Timestamps with timezones: UTC vs local time
Python Solution:
# Try multiple formats
df['date'] = pd.to_datetime(df['date'], format='%m/%d/%Y', errors='coerce')
# Fill failed parses with alternative format
mask = df['date'].isnull()
df.loc[mask, 'date'] = pd.to_datetime(df.loc[mask, 'date'], format='%d/%m/%Y', errors='coerce')

üîó Problem 6: Data Integration Issues

Key Mismatches

Different ID Formats:
  • Problem: customer_id as "CUST001" in one table, 1 (integer) in another
  • Solution: Standardize before joining: df['id'] = df['id'].astype(str).str.zfill(7)
Referential Integrity:
  • Problem: order_id exists in orders table but customer_id doesn't exist in customers table (orphan records)
  • Solution: Use LEFT JOIN and check for nulls, or INNER JOIN to keep only matched records
Granularity Mismatch:
  • Problem: Daily sales data + monthly marketing spend (how to join?)
  • Solution: Aggregate to common level or distribute proportionally

Join Validation

Always Check:
‚Ä¢ Row count before/after: Should match expectations (1:1, 1:many, many:many?)
‚Ä¢ Null count in joined columns: Indicates missing relationships
‚Ä¢ Sum of key metrics: Total revenue shouldn't change after join
‚Ä¢ Duplicate explosion: Did 1:many join create unwanted duplicates?
SQL Validation:
-- Check for orphan records
SELECT COUNT(*) FROM orders o
LEFT JOIN customers c ON o.customer_id = c.id
WHERE c.id IS NULL;

Common Mistakes to Avoid

‚ùå Deleting data without investigation: Always understand WHY data is missing/duplicated before removing.
‚ùå Over-cleaning: Removing all outliers can eliminate legitimate edge cases that are valuable insights.
‚ùå Not saving original data: Always keep raw data untouched. Clean on a copy.

Business Metrics & KPIs Guide

Why This Matters: Knowing SQL/Python makes you a technician. Understanding business metrics makes you a strategic partner. Every industry has critical KPIs that drive decisions.

üéØ Universal Business Metrics

Revenue Metrics

MRR/ARR
Monthly/Annual Recurring Revenue: Predictable revenue from subscriptions.
Formula: Sum of all active subscription values per month/year
Why: Core metric for SaaS, indicates business stability
ARPU/ARPPU
Average Revenue Per User/Paying User: Revenue efficiency per customer.
Formula: Total Revenue √∑ Total Users (or Paying Users)
Why: Tracks monetization effectiveness, segment comparison
GMV
Gross Merchandise Value: Total sales value (marketplace platforms).
Formula: Sum of all transaction values (before fees/returns)
Why: Shows platform scale, different from actual revenue
AOV
Average Order Value: Mean transaction size.
Formula: Total Revenue √∑ Number of Orders
Why: Upselling effectiveness, pricing strategy impact

Growth Metrics

GROWTH RATE
Period-over-Period Growth: Revenue/user expansion speed.
Formula: ((Current - Previous) √∑ Previous) √ó 100
Example: MoM (Month-over-Month), YoY (Year-over-Year), QoQ (Quarter)
CAGR
Compound Annual Growth Rate: Smoothed long-term growth.
Formula: (Ending Value √∑ Beginning Value)^(1/years) - 1
Why: Removes volatility, compares multi-year performance
QUICK RATIO
Growth Efficiency (SaaS): New/expansion vs churned revenue.
Formula: (New MRR + Expansion MRR) √∑ (Churned MRR + Contraction MRR)
Good: > 4.0 (healthy growth), Bad: < 1.0 (shrinking)

üí∞ E-Commerce & Retail Metrics

Conversion Rate
  • Formula: (Purchases √∑ Visitors) √ó 100
  • Benchmarks: E-commerce 2-3%, Retail 20-40%
  • Track by: Source, device, product category, time
  • Improvement levers: UX, pricing, trust signals, checkout friction
Cart Abandonment Rate
  • Formula: (Carts Created - Purchases) √∑ Carts Created √ó 100
  • Industry average: 70% (very high!)
  • Common causes: Unexpected shipping costs, forced account creation, complex checkout
  • Recovery: Email reminders, discount codes, simplified flow
Return Rate
  • Formula: (Items Returned √∑ Items Sold) √ó 100
  • Benchmarks: Clothing 20-30%, Electronics 5-10%
  • Impact: Logistics costs, inventory issues, customer satisfaction
  • Track by: Product, category, supplier to identify quality issues
Inventory Turnover
  • Formula: COGS √∑ Average Inventory Value
  • Higher = better (fast-moving inventory, less capital tied up)
  • Industry varies: Grocery 15-20√ó/year, Furniture 3-5√ó/year
  • Too high: Stock-outs, lost sales. Too low: Waste, obsolescence
Customer Purchase Behavior:
‚Ä¢ Repeat Purchase Rate: % customers who buy again
‚Ä¢ Purchase Frequency: Orders per customer per year
‚Ä¢ Time Between Purchases: Days from first to second order
‚Ä¢ Product Affinity: Items often bought together (cross-sell opportunities)

üì± SaaS & Subscription Metrics

Churn Rate (Customer & Revenue)
  • Customer Churn: (Lost Customers √∑ Starting Customers) √ó 100 per month
  • Revenue Churn: (Lost MRR √∑ Starting MRR) √ó 100 per month
  • Benchmarks: B2B SaaS 5-7% annual, B2C 5-7% monthly (!)
  • Net churn can be negative if expansions exceed losses
CAC (Customer Acquisition Cost)
  • Formula: Total Sales & Marketing Spend √∑ New Customers Acquired
  • Include: Salaries, ads, tools, events, commissions
  • Time period: Typically monthly or quarterly
  • Varies by channel: Organic search (low) vs paid ads (high)
LTV (Lifetime Value)
  • Simple Formula: ARPU √ó Average Customer Lifespan (months)
  • Advanced: ARPU √∑ Churn Rate (assumes constant churn)
  • Example: $50/mo ARPU, 5% monthly churn ‚Üí LTV = $50 √∑ 0.05 = $1,000
  • Critical: LTV should be > 3√ó CAC for healthy business
Payback Period
  • Formula: CAC √∑ (ARPU √ó Gross Margin %)
  • Example: $600 CAC, $100 ARPU, 80% margin ‚Üí 7.5 months payback
  • Target: < 12 months for most SaaS, < 6 months for fast-growth
  • Why matters: Cash flow, growth velocity, investor appeal
NRR (Net Revenue Retention)
  • Formula: (Starting MRR + Expansion - Churn - Contraction) √∑ Starting MRR √ó 100
  • Measures: Revenue retention from existing customers (excluding new)
  • World-class: > 120% (upsells exceed churn)
  • Shows: Product value, expansion success, customer satisfaction
SaaS Magic Number:
Formula: (Current Quarter ARR - Last Quarter ARR) √∑ Last Quarter S&M Spend
Interpretation: > 0.75 = efficient growth, < 0.5 = burning cash inefficiently
Use: Decide when to increase/decrease marketing spend

üìà Advanced Analytics Concepts

Cohort Analysis

What: Group users by shared characteristic (signup month) and track behavior over time
Why: See if retention improves with product changes, identify seasonal patterns
Example: January cohort has 60% month-2 retention vs December's 45% (holiday signups less engaged)
SQL Pattern: GROUP BY signup_month, months_since_signup

RFM Analysis

Recency: How recently did customer purchase? (0-30 days = 5, 31-60 = 4, etc.)
Frequency: How often do they buy? (10+ orders = 5, 7-9 = 4, etc.)
Monetary: How much do they spend? (Top 20% = 5, next 20% = 4, etc.)
Use: Segment customers (555 = VIP, 155 = high-value at risk, 511 = champions)

Funnel Analysis

Setup: Define stages (Visit ‚Üí Signup ‚Üí Trial ‚Üí Purchase ‚Üí Renewal)
Calculate: Conversion rate between each step
Identify: Biggest drop-off (that's your bottleneck!)
Optimize: A/B test improvements at weakest stage
Example: 40% drop at Trial‚ÜíPurchase? Maybe pricing/value prop issue

Correlation vs Causation

Correlation: Two metrics move together (ice cream sales & drownings both up in summer)
Causation: One causes the other (confounding variable: hot weather causes both)
Test causation: A/B test, controlled experiment, regression with controls
Never say: "X caused Y" from correlation alone - always caveat with "associated with"

üéØ Setting Good KPIs

SMART Framework
  • Specific: "Increase trial conversions" not "Grow business"
  • Measurable: "15% conversion rate" not "Better conversions"
  • Achievable: 5% ‚Üí 7% realistic, 5% ‚Üí 50% unrealistic
  • Relevant: Ties to business goal (revenue, retention, satisfaction)
  • Time-bound: "By Q2 2026" not "Eventually"
Leading vs Lagging Indicators
  • Lagging: Revenue, churn (tell you what happened, hard to change quickly)
  • Leading: Trial signups, engagement, NPS (predict future, actionable now)
  • Best practice: Track both - lead with leading, validate with lagging
  • Example: Daily active users (leading) predicts monthly revenue (lagging)
Vanity vs Actionable Metrics
  • Vanity: Total signups, page views, social followers (look good but don't drive decisions)
  • Actionable: Activation rate, revenue per cohort, conversion by channel (inform strategy)
  • Test: Can you make a decision based on this metric? If no, it's likely vanity
  • Example: "1M downloads" is vanity if 90% never open app. "30% D7 retention" is actionable

üö® Common Metric Mistakes

‚ùå Tracking too many KPIs: Focus on 5-7 critical metrics max. More = distraction and analysis paralysis.
‚ùå Ignoring context: 20% churn might be great for B2C gym app, terrible for B2B enterprise SaaS.
‚ùå Not segmenting: Overall conversion rate 3% hides that mobile is 1% and desktop is 8%.
‚ùå Comparing incomparable periods: Comparing holiday season to regular month misleads.
‚ùå Attribution errors: Last-click attribution gives all credit to final touchpoint, ignoring customer journey.
‚ùå Survivorship bias: Only analyzing retained customers misses why others churned.
‚ùå P-hacking: Testing 20 metrics until one shows significance by chance (5% false positive rate).

ü™ü Window Functions (Game Changer)

What: Perform calculations across rows while keeping individual row detail (unlike GROUP BY which collapses rows)
Syntax: function() OVER (PARTITION BY ... ORDER BY ... ROWS/RANGE ...)
Why powerful: Rankings, running totals, moving averages, lead/lag comparisons

ROW_NUMBER, RANK, DENSE_RANK

Use Case: Top N per Group -- Find top 3 products by revenue in each category
WITH ranked_products AS (
  SELECT
    category,
    product_name,
    revenue,
    ROW_NUMBER() OVER (PARTITION BY category ORDER BY revenue DESC) as rn,
    RANK() OVER (PARTITION BY category ORDER BY revenue DESC) as rank,
    DENSE_RANK() OVER (PARTITION BY category ORDER BY revenue DESC) as dense_rank
  FROM products
)
SELECT * FROM ranked_products WHERE rn <= 3;

/* Difference:
ROW_NUMBER: 1,2,3,4,5 (always unique, ties get arbitrary order)
RANK: 1,2,2,4,5 (ties get same rank, next rank skipped)
DENSE_RANK: 1,2,2,3,4 (ties get same rank, no gaps) */

Running Totals & Cumulative Sums

Use Case: Cumulative Revenue by Date SELECT
  date,
  daily_revenue,
  SUM(daily_revenue) OVER (ORDER BY date) as cumulative_revenue,
  AVG(daily_revenue) OVER (ORDER BY date ROWS BETWEEN 6 PRECEDING AND CURRENT ROW) as seven_day_avg
FROM daily_sales
ORDER BY date;

-- ROWS BETWEEN options:
-- UNBOUNDED PRECEDING: From start to current row
-- N PRECEDING: Last N rows including current
-- CURRENT ROW: Just this row
-- N FOLLOWING: Next N rows

LAG & LEAD (Compare Rows)

Use Case: Period-over-Period Growth SELECT
  month,
  revenue,
  LAG(revenue) OVER (ORDER BY month) as prev_month_revenue,
  revenue - LAG(revenue) OVER (ORDER BY month) as absolute_change,
  ROUND((revenue - LAG(revenue) OVER (ORDER BY month)) /
    LAG(revenue) OVER (ORDER BY month) * 100, 2) as pct_change
FROM monthly_revenue
ORDER BY month;

-- LEAD() does opposite: looks forward instead of backward
-- LEAD(revenue, 1) = next month, LEAD(revenue, 3) = 3 months ahead

Use Case: Customer Segmentation by Spend WITH customer_quartiles AS (
  SELECT
    customer_id,

WITH customer_orders AS (

    MAX(order_date) as last_order_date

ROLLUP & CUBE (Subtotals)

Use Case: Hierarchical Totals -- Get totals by year, year+quarter, year+quarter+month
SELECT
  year,
  quarter,
  month,
  SUM(revenue) as total_revenue
FROM sales
GROUP BY ROLLUP(year, quarter, month)
ORDER BY year, quarter, month;

-- ROLLUP creates:
-- (year, quarter, month), (year, quarter), (year), (grand total)

-- CUBE creates all combinations:
-- (year), (quarter), (month), (year,quarter), (year,month),
-- (quarter,month), (year,quarter,month), (grand total)

Use Case: Multiple Metrics in One Query SELECT
  product_id,
  COUNT(*) as total_orders,
  COUNT(*) FILTER (WHERE status = 'completed') as completed_orders,
  COUNT(*) FILTER (WHERE status = 'cancelled') as cancelled_orders,
  SUM(total) FILTER (WHERE date >= '2024-01-01') as ytd_revenue,
  AVG(total) FILTER (WHERE is_first_purchase = true) as avg_first_order_value
FROM orders
GROUP BY product_id;

-- SUM(CASE WHEN status = 'completed' THEN 1 ELSE 0 END)

  country,

üîç Advanced Filtering

HAVING vs WHERE

WHERE: Filters rows BEFORE aggregation (operates on individual rows)
HAVING: Filters groups AFTER aggregation (operates on aggregated results)
Order: WHERE ‚Üí GROUP BY ‚Üí HAVING ‚Üí SELECT ‚Üí ORDER BY
Use Case: Find High-Value, Active Customers SELECT
  customer_id,
  COUNT(*) as order_count,
  SUM(total) as total_spent
FROM orders
WHERE order_date >= CURRENT_DATE - INTERVAL '90 days' -- Recent orders only
GROUP BY customer_id
HAVING COUNT(*) >= 5 AND SUM(total) >= 1000; -- At least 5 orders and $1000 spent

Subqueries in WHERE (IN, EXISTS, ANY, ALL)

Use Case: Find Customers Who Never Purchased -- Method 1: NOT IN (can be slow with NULLs)
SELECT customer_id, email
FROM customers
WHERE customer_id NOT IN (SELECT DISTINCT customer_id FROM orders);

-- Method 2: NOT EXISTS (often faster, NULL-safe)
SELECT c.customer_id, c.email
FROM customers c
WHERE NOT EXISTS (
  SELECT 1 FROM orders o WHERE o.customer_id = c.customer_id
);

-- Method 3: LEFT JOIN with NULL check (most explicit)
SELECT c.customer_id, c.email
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id
WHERE o.customer_id IS NULL;

Correlated Subqueries

Use Case: Find Each Customer's Most Recent Order SELECT
  o.customer_id,
  o.order_id,
  o.order_date,
  o.total
FROM orders o
WHERE o.order_date = (
  SELECT MAX(order_date)
  FROM orders
  WHERE customer_id = o.customer_id -- Correlated: references outer query
);

-- Alternative using window function (usually faster):
WITH ranked_orders AS (
  SELECT *,
    ROW_NUMBER() OVER (PARTITION BY customer_id ORDER BY order_date DESC) as rn
  FROM orders
)
SELECT * FROM ranked_orders WHERE rn = 1;

üöÄ Performance Optimization

Query Execution Order

SQL Written Order: SELECT ‚Üí FROM ‚Üí WHERE ‚Üí GROUP BY ‚Üí HAVING ‚Üí ORDER BY
Execution Order: FROM ‚Üí WHERE ‚Üí GROUP BY ‚Üí HAVING ‚Üí SELECT ‚Üí DISTINCT ‚Üí ORDER BY ‚Üí LIMIT
Why matters: Can't use column alias in WHERE (not evaluated yet), but can in HAVING/ORDER BY

Index Usage Best Practices

Good for Indexes:
  • WHERE clauses: Equality (=) and range (<, >, BETWEEN) on indexed columns
  • JOIN conditions: Index foreign key columns
  • ORDER BY: Index columns you sort by frequently
  • High cardinality: Columns with many unique values (customer_id good, gender bad)
Index Killers:
  • Functions on indexed column: WHERE YEAR(date) = 2024 ‚Üí won't use index
  • Better: WHERE date >= '2024-01-01' AND date < '2025-01-01'
  • Leading wildcards: WHERE name LIKE '%Smith' ‚Üí won't use index
  • Better: WHERE name LIKE 'Smith%' ‚Üí can use index
  • OR conditions: May not use index efficiently, consider UNION instead

EXPLAIN/ANALYZE (Performance Debugging)

Use Case: Identify Slow Queries EXPLAIN ANALYZE
SELECT c.name, COUNT(o.order_id) as order_count
FROM customers c
LEFT JOIN orders o ON c.customer_id = o.customer_id
WHERE c.created_date >= '2024-01-01'

-- "Seq Scan" (bad: full table scan, add index)
-- "Index Scan" (good: using index)
-- High "cost" values (expensive operations)
-- Actual time vs estimated time (mismatched statistics)

Common Optimization Techniques

LIMIT EARLY
Use LIMIT/TOP: If you only need sample or top N, add LIMIT to reduce processing.
FILTER FIRST
WHERE before JOIN: Reduce table size before expensive joins. Use CTEs to pre-filter.
EXISTS vs IN
EXISTS stops early: Faster for large subqueries, stops at first match. IN evaluates all.
PARTITION PRUNING
Use partition key: In WHERE clause for partitioned tables (e.g., date partitions).
MATERIALIZED VIEWS
Pre-compute expensive: Store aggregation results, refresh periodically for reporting.

Why This Matters: Statistical testing separates opinions from facts. It tells you if your findings are real or just random noise. Essential for A/B tests, experiments, and making confident business decisions.

üìä Fundamental Concepts

Hypothesis Testing Framework

The Process:
  • Null Hypothesis (H‚ÇÄ): No effect/difference exists (status quo)
  • Alternative Hypothesis (H‚ÇÅ): Effect/difference exists (what you're testing)
  • Significance Level (Œ±): Threshold for rejecting H‚ÇÄ (typically 0.05 = 5%)
  • P-value: Probability of seeing results this extreme if H‚ÇÄ is true
  • Decision: If p-value < Œ±, reject H‚ÇÄ (result is "statistically significant")
Example: Testing new checkout design
‚Ä¢ H‚ÇÄ: New design has same conversion rate as old (no difference)
‚Ä¢ H‚ÇÅ: New design has different conversion rate (better or worse)
‚Ä¢ Œ± = 0.05 (willing to accept 5% chance of false positive)
‚Ä¢ Run test, get p-value = 0.02
‚Ä¢ Decision: p < 0.05, reject H‚ÇÄ ‚Üí New design IS significantly different

Type I & Type II Errors

TYPE I ERROR
False Positive (Œ±): Reject H‚ÇÄ when it's actually true (found effect that doesn't exist).
Example: Declare new feature improves retention when it doesn't. Launch bad product.
TYPE II ERROR
False Negative (Œ≤): Fail to reject H‚ÇÄ when it's false (missed real effect).
Example: Miss that new feature DOES improve retention. Kill good product.
STATISTICAL POWER
Power = 1 - Œ≤: Probability of detecting effect when it exists (typically want 80%+).
Higher power = less likely to miss real effects. Increase with larger sample size.

Confidence Intervals

What: Range where true value likely falls with X% confidence
Example: "Conversion rate is 5.2% with 95% CI [4.8%, 5.6%]"
Meaning: 95% confident true rate is between 4.8-5.6%
Why better than p-values: Shows magnitude of effect, not just "significant or not"

üß™ A/B Testing (Most Common Use Case)

When to Use A/B Test

Good Use Cases:
  • Testing new feature vs control (landing page, email subject, button color)
  • Comparing two pricing strategies
  • Evaluating algorithm changes (recommendation engine, search ranking)
  • Must have: Clear metric, ability to randomize, sufficient traffic

A/B Test Checklist

1 Define Success Metric
  • Primary metric: Main goal (conversion rate, revenue per user, retention)
  • Secondary metrics: Watch for unintended effects (load time, bounce rate)
  • Guardrail metrics: Must not harm (user satisfaction, error rate)
2 Calculate Sample Size
  • Baseline rate: Current conversion rate (e.g., 5%)
  • Minimum detectable effect (MDE): Smallest change you care about (e.g., 10% relative lift ‚Üí 5.5%)
  • Power: Typically 80% (Œ≤ = 0.20)
  • Significance: Typically 95% (Œ± = 0.05)
  • Use online calculator or formula to get required sample size per variant
3 Randomize Properly
  • 50/50 split (or 90/10 if testing risky change)
  • Random assignment: Hash user_id, not random on each visit
  • Check balance: Both groups should have similar demographics, behavior
  • Avoid contamination: Same user shouldn't switch between variants
4 Run Until Significance
  • Don't peek: Looking at results early increases false positives
  • Run full duration: At least 1-2 weeks to account for day-of-week effects
  • Hit sample size: Must reach calculated sample size for valid results
5 Analyze Results
  • Calculate conversion rates for each variant
  • Run statistical test (two-proportion z-test)
  • Check p-value and confidence interval
  • Verify secondary/guardrail metrics

A/B Test Calculation (Python)

Two-Proportion Z-Test:
import numpy as np
from scipy import stats

# Example data
n_control = 10000 # Users in control
n_treatment = 10000 # Users in treatment
conversions_control = 520 # Conversions in control
conversions_treatment = 580 # Conversions in treatment

# Calculate conversion rates
rate_control = conversions_control / n_control
rate_treatment = conversions_treatment / n_treatment

# Pooled proportion
pooled_p = (conversions_control + conversions_treatment) / (n_control + n_treatment)

# Standard error
se = np.sqrt(pooled_p * (1 - pooled_p) * (1/n_control + 1/n_treatment))

# Z-score
z_score = (rate_treatment - rate_control) / se

# P-value (two-tailed)
p_value = 2 * (1 - stats.norm.cdf(abs(z_score)))

# Confidence interval (95%)
diff = rate_treatment - rate_control
margin = 1.96 * np.sqrt(rate_treatment*(1-rate_treatment)/n_treatment +
                        rate_control*(1-rate_control)/n_control)
ci_lower = diff - margin
ci_upper = diff + margin

print(f"Control: {rate_control:.2%}")
print(f"Treatment: {rate_treatment:.2%}")
print(f"Relative lift: {(rate_treatment/rate_control - 1):.2%}")
print(f"P-value: {p_value:.4f}")
print(f"95% CI for difference: [{ci_lower:.2%}, {ci_upper:.2%}]")

if p_value < 0.05:
    print("‚úì Statistically significant!")
else:
    print("‚úó Not statistically significant")

üìè Choosing the Right Test

Decision Tree

Comparing Two Groups:
‚Ä¢ Continuous data (revenue, time on site) ‚Üí T-test
‚Ä¢ Proportions (conversion, click rate) ‚Üí Z-test
‚Ä¢ Categorical data (satisfied/neutral/unsatisfied) ‚Üí Chi-square
‚Ä¢ Non-normal distribution ‚Üí Mann-Whitney U
Comparing 3+ Groups:
‚Ä¢ Continuous data, normal distribution ‚Üí ANOVA
‚Ä¢ Categorical data ‚Üí Chi-square
‚Ä¢ Non-normal distribution ‚Üí Kruskal-Wallis
Relationships Between Variables:
‚Ä¢ Two continuous variables ‚Üí Pearson correlation
‚Ä¢ Ordinal/ranked data ‚Üí Spearman correlation
‚Ä¢ Predict outcome from features ‚Üí Regression

üî¨ Common Statistical Tests

T-Test (Compare Means)

When to Use:
  • Comparing average values between two groups
  • Example: Average order value for email campaign A vs B
  • Assumptions: Roughly normal distribution, similar variances, independent samples
Python Implementation:
from scipy import stats

# Revenue data for two groups
group_a = [45, 52, 38, 61, 49, 55, 43, 58, 47, 51]
group_b = [62, 58, 71, 65, 69, 63, 67, 70, 64, 68]

# Independent samples t-test
t_stat, p_value = stats.ttest_ind(group_a, group_b)

print(f"Mean A: ${np.mean(group_a):.2f}")
print(f"Mean B: ${np.mean(group_b):.2f}")
print(f"T-statistic: {t_stat:.3f}")
print(f"P-value: {p_value:.4f}")

if p_value < 0.05:
    print("Means are significantly different")
else:
    print("No significant difference")

Chi-Square Test (Compare Proportions)

When to Use:
  • Testing relationships between categorical variables
  • Example: Does product preference differ by age group?
  • Contingency table: Rows = one category, Columns = another
Python Implementation:
from scipy.stats import chi2_contingency
import pandas as pd

# Contingency table: Product preference by age group
data = pd.DataFrame({
    '18-25': [120, 80, 50],
    '26-35': [90, 110, 60],
    '36+': [60, 70, 90]
}, index=['Product A', 'Product B', 'Product C'])

chi2, p_value, dof, expected = chi2_contingency(data)

print(f"Chi-square statistic: {chi2:.3f}")
print(f"P-value: {p_value:.4f}")
print(f"Degrees of freedom: {dof}")

if p_value < 0.05:
    print("Product preference DOES differ by age group")
else:
    print("Product preference does NOT differ by age group")

ANOVA (Compare Multiple Groups)

When to Use:
  • Comparing means across 3+ groups simultaneously
  • Example: Average engagement for 5 different email subject lines
  • Better than multiple t-tests (avoids inflated Type I error)
Python Implementation:
from scipy import stats

# Engagement scores for different campaigns
campaign_a = [4.2, 3.8, 4.5, 4.1, 3.9]
campaign_b = [5.1, 4.9, 5.3, 5.0, 4.8]
campaign_c = [3.5, 3.7, 3.6, 3.8, 3.4]
campaign_d = [4.8, 4.6, 5.0, 4.7, 4.9]

# One-way ANOVA
f_stat, p_value = stats.f_oneway(campaign_a, campaign_b, campaign_c, campaign_d)

print(f"F-statistic: {f_stat:.3f}")
print(f"P-value: {p_value:.4f}")

if p_value < 0.05:
    print("At least one campaign is significantly different")
    print("(Follow up with post-hoc tests to find which pairs differ)")
else:
    print("No significant differences between campaigns")

Correlation Tests

Pearson Correlation (Linear Relationships):
  • Measures strength of linear relationship (-1 to +1)
  • Example: Relationship between ad spend and revenue
  • Assumptions: Both variables continuous, roughly normal, linear relationship
Python Implementation:
from scipy.stats import pearsonr

ad_spend = [1000, 1500, 2000, 2500, 3000, 3500]
revenue = [5000, 7200, 9500, 11800, 14200, 16500]

correlation, p_value = pearsonr(ad_spend, revenue)

print(f"Pearson correlation: {correlation:.3f}")
print(f"P-value: {p_value:.4f}")

# Interpretation:
# 0.7-1.0: Strong positive correlation
# 0.3-0.7: Moderate positive
# 0-0.3: Weak positive
# Negative values: Inverse relationship

‚ö†Ô∏è Common Pitfalls & Mistakes

‚ùå P-hacking: Testing many hypotheses until one shows p < 0.05 by chance. Solution: Pre-register hypothesis, use Bonferroni correction for multiple tests.
‚ùå Peeking: Checking A/B test results before reaching sample size. Solution: Calculate duration upfront, don't look until complete.
‚ùå Confusing significance with importance: p < 0.05 doesn't mean practically meaningful. Solution: Always report effect size and confidence intervals.
‚ùå Sample size too small: Underpowered test misses real effects. Solution: Use power analysis calculator before testing.
‚ùå Assuming normality without checking: T-tests require normal distribution. Solution: Check with histogram/Q-Q plot, use non-parametric test if skewed.
‚ùå Correlation = Causation: Ice cream sales correlate with drownings (both caused by summer heat). Solution: Use experiments/A/B tests for causation, not observational correlation.
‚ùå Ignoring multiple testing problem: 20 tests at Œ±=0.05 means 1 false positive expected. Solution: Bonferroni correction (Œ± / number of tests).

Statistics & A/B Testing Questions

Q: Explain p-value to a non-technical stakeholder
Good answer: "P-value is the probability that our results happened by random chance. If it's below 5% (0.05), we're confident the effect is real, not just luck."
Q: When would you use a t-test vs chi-square test?
Good answer: "T-test compares averages (like mean revenue between two groups). Chi-square tests relationships between categories (like does product preference vary by age group)."

Good answer: "First define success metric (e.g., 7-day retention). Calculate sample size for 80% power to detect 5% lift. Randomly assign 50/50, ensure no contamination. Run for 2 weeks minimum. Analyze with two-proportion test. Check guardrail metrics like engagement and satisfaction."

Good answer: "Statistical significance doesn't equal business significance. I'd calculate the actual business impact (revenue, users affected) and implementation cost. üéì Technical Concept Questions

Q: Explain the difference between LEFT JOIN and INNER JOIN
Good answer: "INNER JOIN returns only rows with matches in both tables. LEFT JOIN returns all rows from the left table, with NULLs for non-matching right table rows. Use LEFT JOIN when you need all records from primary table regardless of matches."
Q: What's the difference between WHERE and HAVING?
Good answer: "WHERE filters rows before aggregation, operates on individual records. HAVING filters after aggregation, operates on grouped results. Use WHERE for row-level filtering, HAVING for filtering aggregated groups."
Q: How do you handle missing data?
Good answer: "Depends on type and amount. First, understand WHY it's missing (MCAR, MAR, or MNAR). If <5% and random, can delete. For numerical: median imputation (skewed) or mean (normal). For categorical: mode or 'Unknown' category. If >40%, investigate data source. Critical: document all decisions and consider creating 'is_missing' flag to preserve signal."
Q: What's correlation vs causation?
Good answer: "Correlation means two variables move together, but doesn't prove one causes the other. Ice cream sales and drownings both correlate because summer causes both. For causation, need controlled experiment like A/B test with randomization to isolate the effect."
Q: How would you detect outliers?
Good answer: "Multiple methods: (1) Visual with box plots, (2) IQR method: outliers beyond Q1-1.5√óIQR or Q3+1.5√óIQR, (3) Z-score: |z| > 3 for normal data, (4) Isolation Forest for complex patterns. Always investigate before removing - could be data error OR important edge case like VIP customer."

Dashboard Design Principles

Why Design Matters: A poorly designed dashboard = ignored insights. Great dashboards are simple, actionable, and answer business questions at a glance. Design is 50% of the value‚Äîdata doesn't speak for itself.

üéØ Dashboard Design Philosophy

Core Principles

CLARITY FIRST
Less is more: Remove everything that doesn't serve a purpose. Every chart should answer a specific question.
Bad: 15 metrics crammed on one page. Good: 5-7 key metrics with context.
AUDIENCE-FOCUSED
Know your user: Executive needs high-level KPIs. Analyst needs drill-down details.
CEO dashboard ‚â† Operations dashboard. Different audiences, different designs.
ACTIONABLE
Enable decisions: User should know what to do after viewing. Insights ‚Üí actions.
Show "Conversion dropped 15%" + "Top issue: mobile checkout errors" + "Fix priority: High"
TRUST & ACCURACY
Build confidence: Show data freshness, sources, methodology. One error destroys credibility.
Include: "Last updated: 2 hours ago" | "Source: Google Analytics" | "Excludes returns"

üìê Layout & Structure

F-Pattern Layout (Eye Movement)

Users scan in F-shape:
‚Ä¢ Top-left: Most important metric (North Star KPI)
‚Ä¢ Top row: Key summary metrics (3-5 cards)
‚Ä¢ Left column: Primary trend charts
‚Ä¢ Center/right: Supporting details, breakdowns
‚Ä¢ Bottom: Granular data, filters, less critical info

Information Hierarchy

Level 1: Executive Summary (Top Section)
  • Big numbers (KPI cards): Current value, % change, trend indicator
  • Example: "Revenue: $1.2M ‚Üë +15% MoM"
  • Limit to 4-6 cards max
Level 2: Trends & Comparisons (Middle Section)
  • Line charts showing performance over time
  • Bar charts comparing segments/categories
  • Allow users to spot patterns and anomalies
Level 3: Details & Drill-Down (Lower Section)
  • Tables with sortable columns
  • Detailed breakdowns by geography, product, channel
  • Filters and interactive elements

Grid System

Use consistent spacing:
‚Ä¢ Align all elements to invisible grid (12-column typical)
‚Ä¢ Consistent margins (20-30px between elements)
‚Ä¢ Whitespace is your friend‚Äîdon't cram
‚Ä¢ Group related metrics visually (proximity principle)

üìä Choosing the Right Chart

Chart Selection Guide

LINE CHART
Use for: Trends over time (daily revenue, monthly users)
Best practice: Start Y-axis at 0, limit to 5 lines max, use direct labels not legend
Avoid: Categorical comparisons (use bar instead)
BAR CHART
Use for: Comparing categories (revenue by product, traffic by channel)
Best practice: Horizontal if long labels, sort by value (descending), label values
Avoid: More than 10-12 bars (use top N + "Other")
KPI CARD
Use for: Single important metric (total revenue, active users)
Best practice: Big number + comparison (vs last period) + sparkline trend
Avoid: Too many decimals (round to 2 sig figs)
STACKED BAR
Use for: Part-to-whole over categories (revenue by product line by region)
Best practice: Limit to 3-4 segments, most important at bottom
Avoid: Comparing segments (hard to compare non-baseline segments)
HEATMAP
Use for: Two dimensions + magnitude (retention cohorts, day-hour patterns)
Best practice: Sequential color scale, clear labels, legend with range
Avoid: Rainbow colors (use single hue gradient)
WATERFALL
Use for: Cumulative effect (revenue changes from Q1 to Q2)
Best practice: Show positive/negative clearly, label each step
Avoid: Too many steps (combine small changes)

Charts to Avoid (Usually)

‚ùå PIE CHART: Hard to compare angles. Use bar chart instead. Exception: 2-3 segments only.
‚ùå 3D CHARTS: Distort perception, look dated. Always use 2D.
‚ùå DUAL-AXIS: Misleading scales. Use separate charts or normalize.
‚ùå RADAR CHART: Difficult to read. Use bar chart for comparisons.
‚ùå DONUT CHART: Worse than pie. Use if showing part-of-whole with central metric.

üé® Color & Visual Design

Color Strategy

Limit Your Palette
  • Primary brand color (for key metric, emphasis)
  • 2-3 supporting colors (for categories, segments)
  • Gray scale (for neutral elements, backgrounds)
  • Semantic colors: Green (good/up), Red (bad/down), Yellow (warning)
  • Total: 5-6 colors maximum across entire dashboard
Color Usage Rules
  • Categorical: Distinct hues (blue, orange, green) for different categories
  • Sequential: Single hue gradient (light blue ‚Üí dark blue) for magnitude
  • Diverging: Two hues from neutral (red-white-blue) for pos/neg variance
  • Accessibility: Test color-blind friendly (use ColorBrewer, avoid red-green only)

‚Ä¢ Primary: #2E86AB (blue) - main metric
‚Ä¢ Success: #06A77D (green) - positive change
‚Ä¢ Warning: #F18F01 (orange) - needs attention

Data Ethics & Privacy

Why This Matters: Data is power. With great power comes great responsibility. As analysts, we access personal information, influence decisions, and shape outcomes. Ethical lapses destroy careers, companies, and lives. This isn't optional‚Äîit's foundational.

‚öñÔ∏è Core Ethical Principles

The Four Pillars

TRANSPARENCY
Be clear about what you're doing: Users should know data is collected, how it's used, who has access.
Example: Privacy policy in plain language, not legal jargon. Cookie consent that's honest, not deceptive.
CONSENT
Ask permission, respect answers: Opt-in, not opt-out. Clear choice without dark patterns.
Bad: Pre-checked boxes, confusing language. Good: Explicit "Yes, use my data" with easy "No" option.
MINIMIZATION
Collect only what you need: Don't hoard data "just in case." Less data = less risk.
Example: Need age bracket? Don't collect exact birthdate. Need location? City is enough, not GPS coordinates.
ACCOUNTABILITY
Take responsibility for impact: When analysis harms, own it and fix it. Document decisions.
Example: Model shows bias? Don't hide it. Investigate, mitigate, disclose limitations to stakeholders.

üîí Privacy Regulations (Must Know)

GDPR (General Data Protection Regulation)

Applies to: EU residents' data (regardless of company location)
Effective: May 2018
Penalties: Up to ‚Ç¨20M or 4% global revenue (whichever is higher)
Key requirements:
GDPR Core Rights:
  • Right to access: Users can request all data you have on them
  • Right to erasure ("right to be forgotten"): Delete user data upon request
  • Right to portability: Provide data in machine-readable format
  • Right to rectification: Correct inaccurate data
  • Right to restrict processing: User can limit how data is used
What Analysts Must Do:
  • Document legal basis for processing (consent, contract, legitimate interest)
  • Implement data retention policies (delete after X years)
  • Pseudonymize/anonymize where possible (replace names with IDs)
  • Enable data export functionality
  • Report breaches within 72 hours

CCPA (California Consumer Privacy Act)

Applies to: California residents, businesses with 25M+ revenue or 50K+ consumers/households
Effective: January 2020
Penalties: $2,500 per violation, $7,500 if intentional
Key difference from GDPR: Opt-out (not opt-in), narrower scope
Consumer Rights:
  • Know what data is collected
  • Delete personal information
  • Opt-out of sale of personal information
  • Non-discrimination for exercising rights

HIPAA (Health Insurance Portability and Accountability Act)

Applies to: Healthcare data in US (medical records, health insurance)
Effective: 1996 (updated 2013)
Penalties: Up to $1.5M per violation type per year
Protected Health Information (PHI): 18 identifiers including name, address, SSN, medical record numbers
Requirements for Analysts:
  • Minimum necessary standard (only access data needed for task)
  • De-identification required for non-essential personnel
  • Business Associate Agreements (BAAs) with vendors
  • Encryption for data at rest and in transit
  • Access logs and audit trails

üé≠ Anonymization vs Pseudonymization

Anonymization (Irreversible)

Definition: Remove all identifying information so individual cannot be re-identified
Techniques: Aggregation, generalization, noise addition, masking
GDPR status: Anonymized data is NOT personal data (fewer restrictions)
Challenge: True anonymization is very difficult (re-identification attacks possible)
Common Techniques:
  • Aggregation: Report only group statistics (avg age, total count), never individuals
  • K-anonymity: Each record indistinguishable from k-1 others (e.g., if k=5, at least 5 people share same attributes)
  • Generalization: Replace specific with general (age 34 ‚Üí 30-40, ZIP 90210 ‚Üí 902**)
  • Noise addition: Add random variation (differential privacy)
  • Suppression: Remove rare combinations entirely

Pseudonymization (Reversible)

Definition: Replace identifiers with pseudonyms (user_id instead of name)
Key: Separate key stores mapping (kept secure, access controlled)
GDPR status: Still personal data, but reduced risk
Use case: Need to track individuals over time but protect identity
Implementation:
  • Hash email addresses: SHA256(email) ‚Üí unique ID
  • Use UUIDs instead of names in analytics database
  • Store mapping table separately with strict access controls
  • Never join pseudonymized data with original identifiers in analysis queries

‚ö†Ô∏è Bias Detection & Mitigation

Types of Bias

SELECTION BIAS
Sample doesn't represent population: Survey only online users misses offline population.
Example: Training hiring model only on current employees (survivorship bias).
MEASUREMENT BIAS
How you measure affects results: Leading survey questions, faulty sensors.
Example: "How much do you love our product?" vs "How satisfied are you with our product?"
CONFIRMATION BIAS
Seeking data that confirms beliefs: Cherry-picking metrics, ignoring contradictory data.
Example: Only highlighting A/B test results when they support desired outcome.
ALGORITHMIC BIAS
ML models perpetuate historical discrimination: Training on biased historical data.
Example: Hiring algorithm favors men because historical hires were 80% male.

Detecting Bias

Checklist for Every Analysis:
  • Is my sample representative? Who's excluded?
  • Are outcomes equal across demographics (gender, race, age)?
  • Does model perform equally well for all groups?
  • What assumptions am I making? Are they valid?
  • Am I only showing data that supports my hypothesis?
Quantitative Tests:
  • Disparate impact ratio: (Group A approval rate) / (Group B approval rate)
  • If < 0.8, likely discriminatory (80% rule)
  • Equal opportunity: False negative rates equal across groups?
  • Calibration: Are predictions equally accurate for all groups?

Mitigation Strategies

Pre-Processing (Before Model):
  • Collect more diverse training data
  • Re-weight samples to balance representation
  • Remove proxy variables (ZIP code can proxy for race)
In-Processing (During Model Training):
  • Add fairness constraints to objective function
  • Use adversarial debiasing techniques
  • Train separate models per group (if appropriate)
Post-Processing (After Model):
  • Adjust decision thresholds per group for equal outcomes
  • Reject predictions with high uncertainty for sensitive groups
  • Regular audits and monitoring of outcomes

üö® Real-World Ethics Failures (Learn From Mistakes)

Case Study 1: Cambridge Analytica (2018)

What happened: Harvested 87M Facebook profiles without consent, used for political targeting
Ethical failures: No consent, misuse of data, manipulation
Consequences: Facebook fined $5B, company dissolved, regulations strengthened
Lesson: Just because you CAN access data doesn't mean you SHOULD use it

Case Study 2: Amazon Hiring Algorithm (2018)

What happened: ML hiring tool penalized resumes with "women's" (trained on 10 years of male-dominated hires)
Ethical failure: Historical bias embedded in model, inadequate testing
Consequence: Tool scrapped, reputational damage
Lesson: Test for bias across protected groups before deployment

Case Study 3: Target Pregnancy Prediction (2012)

What happened: Sent pregnancy-related coupons to teenager; father learned of pregnancy this way
Ethical issue: Revealed sensitive information without consent, invasive prediction
Lesson: Predictive models can reveal private information people haven't disclosed

Case Study 4: Healthcare Risk Algorithm Bias (2019)

What happened: Algorithm used healthcare costs as proxy for health needs, disadvantaged Black patients
Ethical failure: Systemic bias in proxy metric (Black patients have less access ‚Üí lower costs ‚Üí classified as "healthier")
Consequence: Patients denied care they needed
Lesson: Question your proxy metrics‚Äîdo they actually measure what you think?

‚úÖ Ethical Decision Framework

Before Starting Analysis

Ask Yourself:
  • Legality: Do I have legal right to access/use this data?
  • Consent: Did users agree to this specific use case?
  • Necessity: Is this data actually needed, or am I being greedy?
  • Harm potential: Could this analysis hurt individuals or groups?
  • Transparency: Would users be comfortable if they knew about this?

During Analysis

Ongoing Checks:
  • Am I testing for bias across all relevant groups?
  • Are there unintended consequences I'm not considering?
  • Would I be comfortable if my methodology was public?
  • Am I documenting limitations and caveats?
  • Is there someone I should consult (legal, ethics board, domain expert)?

Before Sharing Results

Final Review:
  • Have I disclosed all limitations and assumptions?
  • Could these results be misused? How do I prevent that?
  • Are protected groups represented fairly in insights?
  • Is my recommendation ethical, not just legal?
  • Am I prepared to defend this work publicly?

‚úì Encryption: Use TLS for data in transit, AES-256 for data at rest

‚úì Avoid stigma: Don't label individuals/groups negatively based on data alone

‚úì Empower users: Give people agency over their data (access, deletion, correction)

üéì When to Speak Up

Red Flags That Require Action

üö© Using data without legal basis: "Let's just analyze this data we found"
Action: Verify consent/legal justification before proceeding. Escalate if unclear.
üö© Requested to hide bias/negative findings: "Don't mention the model performs worse for women"
Action: Document everything, insist on disclosure, escalate to ethics committee if needed.
üö© Pressure to cherry-pick data: "Only show the successful segments, ignore the failures"
Action: Present complete picture, note limitations, refuse to mislead stakeholders.
üö© Insufficient security for sensitive data: "Just send me the customer database via email"
Action: Use secure transfer methods, push for proper data governance, report to security team.
üö© Re-identifying anonymized individuals: "Can we figure out who this specific user is?"
Action: Refuse. This violates privacy promises. Suggest aggregate analysis instead.

How to Push Back Professionally

Framework for difficult conversations:
  • Acknowledge: "I understand why you want this insight..."
  • Explain concern: "However, this approach violates GDPR / creates bias / lacks consent..."
  • Offer alternative: "Instead, we could analyze aggregated data / use synthetic data / get proper consent..."
  • Escalate if needed: "I'm not comfortable proceeding. Let's involve [legal/privacy officer/manager]."
